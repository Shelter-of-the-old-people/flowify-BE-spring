Flowify 사용자 경험 요구사항 정의서
1. 서비스 철학
   대상 사용자: 비전문가, 일반인. 자동화나 프로그래밍에 대한 사전 지식이 없는 사용자.

핵심 원칙:

사용자는 "무엇을 원하는지"만 선택한다. 어떤 노드가 되는지는 시스템이 결정한다.
UX 표면에 기술 용어(트리거, 조건 분기, Loop 등)를 노출하지 않는다.
사용자에게 구체적인 기술 설정을 요구하지 않는다. 일상 언어로 질문하고 선택지를 제공한다.
사용자가 서비스 노드를 직접 선택하는 것이 아니라, 사용자의 설정에 따라 노드가 결정된다.
2. 카테고리 및 서비스 체계
   확장성을 열어두고 카테고리 단위로 구성한다. 각 카테고리별로 구체적 서비스를 하나씩 확장해 나간다.

카테고리	서비스 예시	비고
커뮤니케이션	Gmail, Slack
저장소	Google Drive, Notion
스프레드시트	Google Sheets
웹 수집	쇼핑몰, 뉴스, 유튜브	세분화된 하위 카테고리
캘린더	Google Calendar
AI 노드	LLM 기반 처리	중간 과정 전용
시작/도착 노드 선택 시 사용자에게는 카테고리를 먼저 보여주고, 카테고리 선택 후 구체적 서비스를 선택하도록 한다. 사용자가 생각하는 데이터 종류나 용도(문서, 이메일, 쇼핑몰 등) 기준으로 자연스럽게 선택할 수 있도록 한다.

3. 워크플로우 생성 진입점
   두 가지 독립적인 진입점이 존재한다.

3-1. 직접 생성

시작 노드 설정 → 도착 노드 설정 → 중간 과정 설정(직접 또는 AI) 순서로 진행된다.

3-2. 템플릿 기반 생성

별도 템플릿 페이지(/templates)에서 진입한다.

템플릿 리스트 페이지에서 원하는 템플릿을 선택한다.
선택 시 상세보기 페이지로 이동한다. 상세보기란 워크플로우 페이지에 템플릿 정보를 띄운 페이지이다.
"가져오기" 또는 "사용하기" 버튼으로 해당 템플릿을 가져온다.
가져온 템플릿은 사용자의 워크플로우 리스트에 추가된다.
추가된 워크플로우의 편집 화면(/workflows/:id)에서 열린다.
모든 노드가 이미 설정된 상태로 열리며, 사용자가 필요한 부분을 수정할 수 있다.
미인증 서비스가 포함된 경우, 워크플로우는 생성하되 해당 노드에 인증 경고를 표시한다.
4. 직접 생성 흐름
   4-1. 시작 노드 설정
   순서: 카테고리 선택 → 서비스 선택 → OAuth 인증 확인 → 실행 조건 질문 → 구체적 데이터 대상 선택 → 시작 노드 확정

OAuth 인증 확인: 선택한 서비스의 OAuth 인증이 안 되어 있으면, 가장 먼저 뜨는 설정 창이 인증 화면으로 이동하는 설정 창이 된다.

실행 조건 질문: "트리거 설정"이 아닌 "언제 실행할까요?" 같은 일상 언어로 질문한다. 선택지는 서비스에 따라 달라진다.

예시 — Google Drive의 경우:

"파일 사용" (수동/즉시 실행)
"특정 파일이 변경되었을 때" (이벤트 트리거)
"특정 폴더에 파일이 들어올 때" (이벤트 트리거)
사용자의 선택에 따라 내부적으로 트리거 타입(수동, 이벤트, 스케줄)이 결정된다.

구체적 데이터 대상 선택: 어떤 데이터를 사용할 것인지 선택한다. (예: 특정 폴더, 특정 파일, 특정 메일함 등)

4-2. 도착 노드 설정
순서: 카테고리 선택 → 서비스 선택 → OAuth 인증 확인 → 동작 유형 선택

시작 노드와 동일한 방식으로 진행된다. 동작 유형 예시: Notion 페이지 생성, Slack 메시지 전송, Google Drive에 파일 저장 등.

세부 설정(어떤 페이지에, 어떤 형식으로, 어떤 채널에 등)은 이 시점에서 진행하지 않는다. 중간 과정이 모두 완성되어 들어오는 데이터가 확정된 후 마지막에 진행한다.

4-3. 중간 과정 결정
시작 노드와 도착 노드가 확정된 후, "생성 방식을 결정하세요" 화면이 표시된다.

방식 A — AI로 중간 과정 생성하기

채팅형 대화 인터페이스로 진행된다.
AI가 사용자에게 구체적인 요구사항을 질문하며 수집한다.
AI는 직접 설정 기능 전체를 학습한 상태이며, 노드 구성과 설정을 직접 수행한다.
노드 설정값, 트리거 방식 등 구현에 필요한 모든 정보가 해소되면 "충분하다"고 판단한다.
완료 시 완성된 워크플로우 시각화(노드 연결 형태)를 바로 표시한다.
사용자가 직접 설정과 완전히 동일한 방식(노드 클릭 → 양 옆 설정 창)으로 수정할 수 있다.
개발 순서: 직접 설정 기능을 먼저 구현하고, 이를 AI에게 학습시켜 구현한다.
방식 B — 직접 설정하기

아래 4-4에서 상세히 기술한다.

(검토 사항) 직접 설정 도중 AI 생성으로 전환하는 기능은 설계 단계에서 검토한다.

4-4. 직접 설정 상세
화면 구조:

워크플로우 전체가 노드 연결 형태로 화면에 표시된다.
미설정 위치에 "+" 버튼과 "다음 노드를 설정하세요" 텍스트가 표시된다.
"+" 버튼을 누르거나 기존에 설정된 노드를 선택하면, 양 옆에 설정 창이 표시된다.
왼쪽: 이전 노드에서 들어오는 데이터 + 설정 선택지
오른쪽: 현재 노드에서 처리되어 다음으로 넘어가는 데이터
완성된 워크플로우에서도 노드 클릭 시 동일한 설정 창이 열려 수정이 가능하다.
설정 흐름 패턴:

각 노드 설정은 다음 패턴으로 진행된다.

1차 — 처리 방식 선택. 모든 데이터 타입에 공통으로 적용된다. 목록형 데이터의 경우 "한 건씩" 또는 "전체 사용"을 먼저 묻는다. "한 건씩" 선택 시 내부적으로 Loop 노드가 된다.

2차 — 데이터 타입에 따라 다른 선택지 목록이 표시된다. 분류/분기 선택지, 가공 선택지, 기타 동작 선택지가 함께 표시된다. 추천되는 선택지는 순서를 위로 올려서 표현한다. 사용자의 선택이 해당 노드의 타입을 결정한다:

분류 기준 선택 (예: "파일 종류별로", "발신자별로") → 조건 분기 노드가 됨
가공 선택 (예: "내용 요약/정리", "번역") → AI 노드 또는 기타 처리 노드가 됨
기타 동작 → 해당하는 노드 타입으로 결정됨 (구체적 목록은 시나리오 확보 후 도출)
3차 — 후속 설정. 2차에서의 선택에 따라 후속 설정 화면이 결정된다. 기본적으로 사용자가 선택할 수 있는 선택지를 제공하고, 사용자 직접 입력란은 마지막 설정 단계에서만 제공한다.

노드 타입 매핑 종합:

사용자 선택	내부 노드 타입
"한 파일씩" / "하나씩"	Loop 노드
"파일 종류별로" / "발신자별로" 등 분류 기준	조건 분기 노드
"내용 요약/정리" / "번역" 등 가공	AI 노드 또는 기타 처리 노드
하나의 노드에서 복합 데이터가 동시 출력	분기 노드
데이터 타입 변환: 노드 처리 결과에 따라 데이터 타입이 변환될 수 있다. 예: 파일 목록 → AI 요약 → 텍스트. 다음 노드 설정 시 변환된 데이터 타입 기준으로 새로운 선택지가 제공된다.

조건 분기 후 흐름: 분기된 각 경로마다 독립적인 "+" 버튼이 표시되고, 각 경로별로 독립적인 설정 흐름이 진행된다. 모든 분기 경로가 동일한 도착 노드로 합류할 수 있다. 일부 분기 경로만 먼저 설정하고 나머지는 나중에 설정할 수도 있다.

워크플로우 종료: 사용자가 "+" 대신 도착 노드를 직접 선택하면, 들어오는 데이터에 맞게 도착 노드의 세부 설정 화면이 열린다. 설정 완료 시 해당 경로의 워크플로우가 완성된다.

호환성 검증: 중간 과정의 출력 데이터가 도착 노드의 입력과 호환되지 않는 경우 사용자에게 경고를 표시한다.

이전 단계 변경: 사용자가 이전 단계로 돌아가서 선택을 변경할 수 있다. 이후에 이미 설정된 노드들은 삭제하지 않고 유지하되, 호환성 경고를 표시한다.

노드 삭제: 노드를 삭제하면 해당 노드 이후의 모든 노드가 함께 삭제된다. 이후 노드가 이전 노드의 출력에 의존하므로 유지할 의미가 없기 때문이다. 조건 분기 노드를 삭제할 경우, 모든 분기 경로의 후속 노드가 함께 삭제된다.

5. 데이터 타입 체계
   선택지 매핑의 기준이 되는 데이터 타입 분류이다.

데이터 타입	설명
파일 목록	여러 파일이 들어오는 경우
단일 파일	하나의 파일이 들어오는 경우
텍스트	AI 요약 결과 등 텍스트 데이터
스프레드시트 데이터	시트/셀 단위의 구조화된 데이터
이메일 목록	여러 이메일이 들어오는 경우
API 응답 데이터	JSON 형식의 외부 API 응답
API 응답 데이터 처리: JSON 응답을 시스템이 미리 매핑하여 필요한 데이터만 추출한다. 추출 대상 예시: 상품명, 평점, 가격, 특정 키워드 등. 웹 수집의 경우 크롤링이 필요할 수 있으므로, 대상 페이지별로 HTML 태그를 미리 정해두어 데이터를 추출한다. 이 크롤링 규칙은 개발팀이 서비스 코드 레벨에서 미리 구현해둔다.

데이터 타입 변환: 노드 처리 결과에 따라 데이터 타입이 변환될 수 있다. 다음 노드 설정 시 변환된 데이터 타입 기준으로 선택지가 제공된다.

6. 선택지 매핑 시스템
   목적: 각 데이터 타입별로 어떤 선택지를 보여줄지 매핑하는 규칙 체계.

특성:

선택지는 트리 구조가 아니다. 서버에서 미리 정해둔 선택지가, 사용자가 선택하는 노드나 설정에 따라 동적으로 다른 설정창을 제공한다.
데이터 타입별 매핑이다. 이전 노드에서 어떤 형식의 데이터가 들어오는지에 따라 설정 화면이 달라진다.
백엔드에서 동적으로 제공한다.
도출 순서:

각 팀원이 자유로운 형식으로 다양한 자동화 시나리오를 충분히 확보한다.
시나리오들을 분석하여 데이터 타입별 선택지 매핑 규칙을 도출한다.
매핑 시스템의 관리 방식을 설계한다.
관리 방식: 4/1 설계 발표 전 결정 필요. 미정.

7. 자동화 시나리오 및 템플릿
   각 팀원이 본인이 원하는 자동화 시스템을 구축하고, 각 단계별로 노드로 분해하여 서비스에 추가한다.

시나리오의 형식은 정해두지 않는다. 형식을 제한하면 특정 용도에 맞춘 서비스가 되므로, 범용적 서비스 구현이라는 목적에 맞게 자유 형식으로 진행한다.
시나리오에서 선택지 매핑 규칙을 도출하고, 시나리오 자체가 템플릿이 된다.
다양한 자동화 시나리오를 확보하는 것이 중요하다.
8. 실행 화면
   편집 화면과 동일한 노드 연결 시각화를 사용한다. 실행 중 각 노드의 실행 상태를 실시간으로 함께 표시한다.
